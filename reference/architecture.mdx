## Overview

The skill is structured as an **orchestrator + agents** pattern. `SKILL.md` acts as the orchestrator: it classifies the user's intent and routes to a specialized agent file. Each agent file contains its own workflow, SQL queries, and formatting instructions.

This design keeps concerns separated — the orchestrator handles routing logic, while agents handle implementation details.

## Directory structure

```
imessage-mail/
├── SKILL.md                    # Orchestrator — intent classification and routing
└── agents/
    ├── recent-activity.md      # Recent contacts and email overview
    ├── conversation-reader.md  # Full message history for a contact
    ├── message-search.md       # Cross-source keyword search
    ├── attachment-finder.md    # Find files and media from messages/emails
    ├── mail-reader.md          # Apple Mail database queries
    └── send-message.md         # Send iMessages and emails via AppleScript
```

## Intent classification

The orchestrator maps input patterns to agent files:

| Input pattern | Agent file | Description |
|---|---|---|
| "recent messages", "who texted me", "latest texts" | `agents/recent-activity.md` | List recent unique iMessage contacts with names |
| Contact name or phone number + "messages"/"texts" | `agents/conversation-reader.md` | Read iMessage history for a specific contact |
| "find attachment", "photos from", "files from" | `agents/attachment-finder.md` | Find attachments in iMessage conversations |
| "search for", "find messages about", keyword search | `agents/message-search.md` | Full-text search across all iMessages |
| "email", "mail", "inbox", "unread emails" | `agents/mail-reader.md` | Read, search, and inspect Apple Mail emails |
| "send", "text", "reply", "respond", "tell them" | `agents/send-message.md` | Send iMessages or emails via AppleScript |

## Execution flow

<Steps>
  <Step title="Classify intent">
    The orchestrator reads the user's input and matches it against the intent classification table. Ambiguous inputs (e.g., "messages from John") may spawn parallel subagents for both iMessage and Mail.
  </Step>
  <Step title="Read the agent file">
    The orchestrator reads the matched agent `.md` file from `~/.claude/skills/imessage-mail/agents/`. The agent file contains the full workflow, SQL queries, and output format instructions.
  </Step>
  <Step title="Run SQL or AppleScript">
    The agent executes SQLite queries via `sqlite3` against local macOS databases, or AppleScript via `osascript` for send operations. Contact resolution queries the Address Book database.
  </Step>
  <Step title="Return formatted results">
    Results are presented in clean tables or conversational format. iMessage and Mail results are kept in separate sections unless the user requests a combined chronological view.
  </Step>
</Steps>

## Input detection logic

The orchestrator applies the following rules in order when classifying intent:

1. **Send/reply keywords** — If input contains "send", "text [someone]", "reply", "respond", "tell them", "message [someone] saying", route to `send-message`.
2. **Email/mail keywords** — If input contains "email", "mail", "inbox", "unread email", "newsletter", or `@` addresses, route to `mail-reader` (unless send intent is also present, then `send-message`).
3. **Phone number detection** — If input contains a phone number pattern (digits, `+`, dashes, parentheses), check for send intent first; otherwise extract digits and route to `conversation-reader`.
4. **Attachment keywords** — If input contains "attachment", "photo", "image", "file", "video", "pdf", route to `attachment-finder`.
5. **Search keywords** — If input contains "search", "find messages", "look for", route to `message-search`.
6. **Recent/activity keywords** — If input contains "recent", "latest", "who texted", "new messages", route to `recent-activity`.
7. **Contact name** — If input is a person's name (possibly with "messages"/"texts"), route to `conversation-reader` (resolve the name first via Address Book).
8. **Ambiguous "messages from [name]"** — If unclear whether the user means iMessage or email, spawn `conversation-reader` and `mail-reader` subagents in parallel, then combine results.
9. **Default** — Route to `recent-activity`.

## Subagent patterns

The orchestrator uses parallel subagents in three situations to improve performance:

**Parallel contact resolution** — When multiple phone numbers need to be resolved to names, they are batched into 2–3 Agent tool calls (each resolving 5–7 numbers) that run simultaneously rather than one at a time.

**Cross-source search** — When searching for messages from a specific person, two subagents run in parallel: one searches iMessage via `conversation-reader`, the other searches Mail via `mail-reader`. Results are merged and presented chronologically.

**Rich text extraction batch** — When multiple messages show `[Rich Text/Hidden]` content (stored as `NSAttributedString` blobs in `attributedBody`), a single subagent extracts all of them in one batch rather than querying one message at a time.
